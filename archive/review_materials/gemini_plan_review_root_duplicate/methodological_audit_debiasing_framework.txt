Module 1: Acoustic Confounding & Adversarial De-Biasing
Architecture
Crowdsourced respiratory audio datasets exhibit wide device and environmental variability. For
example, the Coswara data collection explicitly records each participant’s device type and ambient
environment along with demographics【4†L630-L639】. Its manual quality audit found ~10% of
recordings to have significant noise (“poor” quality)【4†L724-L733】. Similarly, the COUGHVID corpus is
global and crowdsourced; it includes a 51-column metadata file with fields such as an expert “quality”
label (good/ok/poor) and estimated SNR for each recording【14†L848-L856】【14†L904-L913】. These
metadata fields make clear that confounding artifacts – smartphone microphone frequency responses,
variable sampling rates, room reverberation, background noise and compression codecs – vary
systematically across samples. For instance, COUGHVID audio was captured via web browsers in
compressed Opus (.webm) format (metadata shows “encoder: Chrome” in example data【14†L848L856】), whereas Coswara samples come from diverse smartphones at different sample rates. Even
speaker accent and language (Coswara is mostly Indian languages; COUGHVID spans many regions)
can correlate with COVID labels, risking spurious classification signals.
To de-bias against such confounders, we adopt an adversarial multi-task learning framework with a
gradient reversal layer (GRL)【19†L143-L151】【20†L7-L15】. Let xi be a multi-modal audio input
(cough, breath, speech triplet), yi the true respiratory condition (e.g. COVID vs healthy), and ci a
confounder label (e.g. device type, noise category, or demographic group). A shared encoder fθ (x)
produces latent features z = fθ (x). Two classifier heads operate on z : a respiratory classifier gϕ (z)
(predicting y ) and a confounder classifier dψ (z) (predicting c). We define the clinical loss and confound
loss as

Module 2: Self-Supervised Audio Features & Transformer-Based
Fusion
We eschew handcrafted MFCCs or simple spectrogram CNNs and instead leverage frozen selfsupervised audio encoders. For each modality (Cough, Breath, Speech), we pass the raw waveform
through a pretrained transformer-based backbone (e.g. Wav2Vec 2.0 or HuBERT) to obtain a highdimensional latent embedding sequence. For example, Wav2Vec 2.0 uses a convolutional encoder to
map raw audio X to a sequence of latent vectors z1 , … , zT , which are then input to a Transformer for
contextualization【50†L123-L131】. In our implementation, we load such a model from HuggingFace
(e.g. facebook/wav2vec2-base ), pass each audio channel through it in evaluation mode (no
gradient), and extract the last hidden layer representations Hc , Hb , Hs for cough, breath, speech
respectively. Each Hm is a 2D tensor of shape (time steps × feature dim).
To fuse modalities, we replace naive averaging with cross-modal attention. We apply Transformer-style
multi-head cross-attention layers so that each modality can attend to the others. Concretely, we linearly
project each modality’s features into query, key, and value spaces: e.g.

Qc = Hc W Q ,

Kb = Hb W K ,

Module 3: Cross-Dataset Covariate-Shift Validation
We define a rigid cross-dataset evaluation protocol to quantify out-of-distribution (OOD) performance
drops. Concretely, we train the full model on one dataset and test it on the other without fine-tuning.
For instance, train on Coswara and compute predictions on a held-out COUGHVID test set, and vice
versa. We measure standard metrics (accuracy, AUC, sensitivity, specificity, etc.) for in-dataset evaluation
(e.g. train on Coswara / test on Coswara) and for cross-dataset evaluation (train Coswara / test
COUGHVID). The absolute generalization drop is defined as the difference between in-dataset and
cross-dataset scores. For example, if Mintra = 0.90 and Mcross = 0.65, the drop is Δ = 0.25. (We
report Δ for each metric.) We also report relative drop Δ/Mintra . All results should include confidence
intervals or significance testing (e.g. by bootstrapping) to ensure rigor.
This evaluation is mandatory (not optional). Recent work confirms that even high-performing cough
classifiers can degrade significantly across datasets【33†L99-L105】. For example, Islam et al. (2026)
evaluated COVID cough models across five different datasets and found “demographic and geographic
differences in the cough sounds” that challenge generalization【33†L99-L105】. Our protocol explicitly
mirrors this: it quantifies how performance collapses when encountering “completely unmapped”
recording conditions. By treating Coswara and COUGHVID as separate domains, we force the model to
reveal its true robustness. All numeric results will be tabulated (e.g. with a matrix showing train/test
pairs and metric differences) and visualized (e.g. bar plots of performance drop).

Module 4: Uncertainty Calibration Audit
Beyond accuracy, we evaluate probability calibration both in-domain and under cross-dataset shift.
We apply standard post-hoc calibration methods (Temperature Scaling and Isotonic Regression) on a
validation split of each training dataset, and then measure their effect on both held-out same-domain
and cross-domain test sets. Literature shows that covariate shift breaks calibration: a model calibrated
on source data can become miscalibrated on a shifted target【35†L223-L228】. Specifically, Guo et al.
(2017) temperature scaling assumes i.i.d. data – when this is violated, the mapping no longer aligns
predicted confidence with true probability. Our framework quantifies this calibration degradation using
Expected Calibration Error (ECE) and Maximum Calibration Error (MCE)【37†L578-L584】.

We also assess the clinical risk of miscalibration. A key metric is the frequency of overconfident errors:
for example, the average confidence among wrong predictions on the OOD set. Intuitively, a model
outputting e.g. 99% confidence when incorrect is dangerous. We will report confusion matrices and
reliability diagrams for both domains. As a concrete illustration, Cearns et al. (2020) show a case where
an uncalibrated model gave a 10% predicted probability of response (implying 90% confidence of
failure) but after calibration it gave 47%【48†L238-L247】. In that example the clinician changed course.

Module 5: Implementation Roadmap (PyTorch + HuggingFace)
We outline a step-by-step plan for coding this framework, suitable for a Codex-like implementation
agent:
• (a) Data Handling: Implement a torch.utils.data.Dataset that yields triplets. Using the
metadata tables, match each subject’s Cough, Breath, and Speech recordings into one example.
The __getitem__ should load the raw audio files (e.g. via torchaudio.load ) and return
(waveform_cough, waveform_breath, waveform_speech, label, confound_id) . The
confound_id encodes e.g. device or noise category. Ensure normalization: zero-mean unitvariance or standardized sampling rate (e.g. resample all to 16 kHz if needed). Use
DataLoader for batching.
• (b) SSL Backbone Hooks: Use HuggingFace Transformers. For example, load a pretrained
Wav2Vec2 model and processor:

from transformers import Wav2Vec2Processor, Wav2Vec2Model
processor = Wav2Vec2Processor.from_pretrained('facebook/wav2vec2-base')
model_w2v = Wav2Vec2Model.from_pretrained('facebook/wav2vec2base').eval().to(device)

For each input waveform, apply
processor(waveform, sampling_rate, return_tensors='pt') then model_w2v(**inputs)
to get .last_hidden_state (shape [T, D] ). These features are detached from the graph (no
gradient). Do this for cough, breath, speech independently. Stack or keep them separate as needed.
• (c) Cross-Attention Fusion Module: Define PyTorch nn.Module for the fusion block. For
example, one can use three multi-head attention layers (c->b&s, b->c&s, s->c&b) and appropriate
projections. Pseudocode:

# Cough attends to Breath

• (d) Loss and Training: After fusion, pass the fused vector through the classification head g_φ
(e.g. an MLP) to predict the respiratory label, and through an adversarial confound head d_ψ to
predict the confound_id. Compute loss_clinical = CrossEntropyLoss(y_pred,
y_true) and loss_confound = CrossEntropyLoss(c_pred, c_true) . Insert a gradient
reversal function before the confound head so that when backpropagating loss_confound ,
the encoder’s weights are updated with inverted gradient. In code, one can implement GRL as a
custom autograd Function or use libraries like torchrevgrad . The combined loss is
loss_total = loss_clinical - λ * loss_confound . In each training step, do:
optimizer.zero_grad()
L = loss_clinical - lambda_val * loss_confound
L.backward()
optimizer.step()
This realizes the objective Ltotal . Ensure that only the confound head is updated to minimize
loss_confound , while the encoder is updated to maximize it (via GRL).
• (e) Validation and Logging: As the model trains, log in-dataset and cross-dataset accuracy, as
well as ECE/MCE. Use a held-out split of Coswara to tune temperature for calibration, and
similarly for COUGHVID. After each epoch, evaluate on the opposite dataset without adaptation
to monitor OOD performance. Save model checkpoints based on combined criteria (e.g. best
cross-dataset AUC).
• (f) Hyperparameters & Reproducibility: Document all hyperparameters (learning rate, batch
size, λ value, number of attention heads, etc.) and use fixed random seeds. Prepare scripts to
fully reproduce experiments.
Each component above should be modular and extensively commented. This roadmap provides a clear
blueprint: (a) a triplet Dataset class to fetch matched cough/breath/speech waveforms; (b)
HuggingFace-backed feature extraction calls; (c) a PyTorch module implementing the cross-attention
fusion as sketched; (d) the dual-head classifiers with GRL and losses; (e) the training loop integrating all
parts. Together, they constitute a rigorously specified pipeline ready for implementation.
